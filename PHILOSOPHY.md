# PHILOSOPHY — 왜 이 구조인가

AI 에이전트를 "강력한 개인 동료"로 가정하고 맡기면, 일정 규모 이상의 프로젝트에서는 반드시 문제가 생긴다.
이 리포지토리는 그 문제를 **구조적으로 억제**하려는 실험의 결과물이다.

---

## 전제: AI 에이전트의 3가지 실패 모드

| 실패 모드 | 증상 | 해결 방향 |
|----------|------|----------|
| **컨텍스트 공백** | 매 세션 제로에서 시작. "이 프로젝트는 이렇게 짠다"라는 암묵지가 없다 | 규칙을 **문서로 명시** → `docs/` |
| **작성자 편향** | 자기가 쓴 코드의 문제를 자기가 검토하면 놓친다 | **역할 분리** → 작성자(A) ↔ 검토자(B) 서브에이전트 |
| **이탈** | 허용되지 않은 파일 수정, 스코프 밖 리팩토링, 정책 위반 | **도구 권한 제한**과 **오케스트레이션** → `.claude/` 전반 |

---

## 설계 원칙

### 1. 규칙은 추측 대상이 아니라 **참조 대상**이다

에이전트가 "이 프로젝트 스타일은 아마 이럴 거야"라고 추론하는 순간, 실패가 시작된다.
모든 규칙을 **문서화하고**, 에이전트가 필요한 문서만 골라 읽게 한다.

- `docs/backend/architecture/*` — 레이어별 코딩 규칙, 네이밍, 파일 구조, **Post-Work Verification 체크리스트**
- `docs/backend/policies/*` — 레이어를 관통하는 정책 (멀티테넌트, DDL 동기화, 보안, 로깅)
- 각 서브에이전트는 자신의 역할에 필요한 문서만 "Source of Truth"로 지정받는다

### 2. 작성자는 검토자가 될 수 없다

자기 코드를 스스로 검토하는 에이전트는 **자기가 왜 그렇게 짰는지**에 끌려다닌다.
그래서 설계(D)·작성(A)·검토(B)는 **다른 서브에이전트 인스턴스**로 완전히 분리한다.

- **Agent D (`design-writer`)** — 코드 작성 전에 TDD(기술설계문서) 작성. 코드 수정 금지.
- **Agent A (`code-writer`)** — 코드 작성·수정·테스트. 자기검증 금지.
- **Agent B (`architecture-reviewer`)** — 문서 대조·위반 보고. `Edit`/`Write` 도구 없음(원천 차단).
- **D·A·B는 서로를 호출하지 않는다.** 모든 통신은 메인 오케스트레이터(`implement` 스킬)를 경유.

### 3. 피드백 루프는 "다음에 잘하기"보다 **"지금 수렴하기"** 다

사람은 "다음부터 주의하겠다"로 개선하지만, 에이전트는 다음 세션에 이전 맥락을 잃는다.
따라서 개선은 **지금 이 작업 안에서** 일어나야 한다.

`implement` 스킬은 각 마일스톤마다 다음 루프를 돈다:

```
D(설계) → A(구현) → B(검토) →
  PASS  → 다음 마일스톤
  위반  → A(수정) → B(재검토) → ... (최대 5회)
  5회↑ → 사용자 escalate
```

### 4. 도구 권한은 "할 수 있는 것"이 아니라 **"할 수 없는 것"** 으로 정의한다

에이전트가 "실수할 수 있는 범위"를 도구 권한으로 원천 차단한다.

- `architecture-reviewer`는 `Read/Glob/Grep`만 보유 → 파일 수정 자체가 불가능
- `/implement` 커맨드는 `allowed-tools`로 사용 가능한 도구를 화이트리스트로 지정
- 위험 명령은 `settings.json`의 권한 시스템에서 사용자 승인 필요

### 5. 암묵지는 **맥락 이식 가능한 형태**여야 한다

프로젝트를 다른 팀·다른 사람이 이어받을 때, 암묵지가 코드에 녹아 있으면 전달이 불가능하다.
모든 결정의 "왜"가 `docs/`에 남도록 설계한다. 이 리포지토리 자체도 그 원칙을 따른다.

## 이 철학이 만들어내는 행동 변화

### Before (가드레일 없음)
- 에이전트가 요구사항을 받고 즉시 코드를 쓰기 시작
- 설계 없이 구현하다 방향이 틀려 대규모 재작업 발생
- 아키텍처 규칙을 어겨도 스스로 알아채지 못함
- 리뷰어(사람)가 매번 동일한 위반을 지적
- 작업 결과가 에이전트 세션 종료와 함께 암묵지에서 사라짐

### After (이 템플릿)
- 에이전트가 요구사항을 마일스톤으로 분해, 사용자 확인 후 진행
- 각 마일스톤마다 **D가 TDD 작성** → **A가 TDD 기반 구현** → **B가 문서 기반 검토**
- 위반 발견 시 자동 수정, PASS까지 수렴
- 규칙은 `docs/`에 축적, 다음 세션도 동일하게 동작
