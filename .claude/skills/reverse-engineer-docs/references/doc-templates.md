`{중괄호}` 항목은 반드시 실제 값으로 교체한다. 해당 정보가 없으면 `> [분석 필요: {이유}]`로 표시한다.

## architecture/ 문서 작성 스타일

`architecture/` 하위 문서(모듈 가이드라인·컨벤션)는 아래 4가지 원칙을 반드시 따른다.

**1. 판단 기준을 준다**
"언제 이 컴포넌트를 쓰고 언제 쓰지 않는가"를 명시한다. 모호한 경우를 포함하여 실제 판단이 필요한 상황을 서술한다. 원칙이나 규칙만 나열하고 끝내지 않는다.

**2. 불릿 형식**
원칙·책임·판단 기준은 불릿(`-`) 목록으로 서술한다. 항목당 한 문장이 기본이며, 이유나 결과를 덧붙일 때는 같은 불릿 안에서 이어쓴다. 테이블은 네이밍 패턴처럼 구조적 비교가 꼭 필요한 곳에만 쓴다.

**3. 추상적 원칙·책임·판단 기준을 서술형으로 작성**
"왜 이 계층이 이 일을 담당하는가", "이 컴포넌트가 담당하지 않는 것은 무엇인가", "어떤 상황에서 다른 컴포넌트를 선택하는가"를 서술형 문장으로 쓴다. 단순 레이블("책임: 조회")이 아니라 맥락이 있는 문장으로 표현한다.

**4. 선언형 규칙**
규칙은 "X는 Y다", "A는 B에 위임한다", "C에 D가 들어오면 E가 의미를 잃는다" 형태의 선언형 문장으로 쓴다. "X를 하세요", "Y처럼 작성합니다" 같은 명령형·경어체를 쓰지 않는다.

## 3. 모듈 가이드라인 문서

파일명 패턴: `{layer}-module-guidelines.md`

- **원칙**: 이 계층이 존재하는 이유와 지켜야 할 불변식을 선언형 문장으로. 위반 시 생기는 문제를 같은 불릿에 이어 쓴다.

## 4. 컨벤션 문서

파일명 패턴: `{concept}-convention.md`

각 섹션의 작성 방식:
- **원칙**: 이 컴포넌트가 지켜야 할 불변식을 선언형 문장으로. "A는 B다", "C가 D에 들어오면 E가 의미를 잃는다" 형태. 이유를 같은 불릿 안에서 이어 쓴다.
- **핵심 규칙**: 가장 중요한 불문율 하나를 굵게 선언한다. 절대 두 개 이상으로 늘리지 않는다.
- **책임**: 이 컴포넌트가 담당하는 것과 담당하지 않는 것. 어디에 위임해야 하는지까지.
- **판단 기준**: 이 컴포넌트를 만들어야 하는 상황과 만들지 말아야 하는 상황을 불릿으로. 판단이 흔들리는 경계 케이스를 특히 명시한다.
- **금지 사항**: "X하지 않는다" 형태의 선언형 금지 규칙.

```markdown
# {개념명} 컨벤션

## 원칙

- {선언형 원칙}. {이 원칙의 이유 — 위반 시 어떤 문제가 발생하는가.}
- {선언형 원칙}. {코드베이스에서 발견한 근거 또는 이 컴포넌트의 존재 이유.}
- {선언형 원칙}.

---

## 핵심 규칙

**{이 컴포넌트의 가장 중요한 불문율을 선언형 한 문장으로. "A는 B다" 또는 "A는 B에 위임하고 C만 결정한다."}**

{위 규칙이 지켜지지 않으면 발생하는 결과를 1~2문장으로 보충한다.}

---

## 책임

- {이 컴포넌트가 담당하는 것 — 어떤 결정을 내리고 어떤 행위를 하는가}
- {이 컴포넌트가 담당하는 것}
- **담당하지 않는 것**: {이 컴포넌트에 들어오면 안 되는 로직. 어느 컴포넌트에 위임해야 하는가.}

- {이 컴포넌트를 만들어야 하는 상황 — 어떤 조건이 충족될 때}

## 6. design/README.md

```markdown
# Design Documents (TDD)

{프로젝트명} 기술 설계 문서(Technical Design Document) 작성 규칙.

---

## 언제 작성하는가

다음 중 하나에 해당하면 TDD를 먼저 작성한다:

- 새로운 도메인 개념 또는 애그리거트를 도입할 때
- 기존 계층 구조를 횡단하는 복잡한 플로우를 구현할 때
- 트랜잭션 경계나 동시성 제어 설계가 필요할 때
- 외부 시스템과 새롭게 통합할 때

**작성하지 않는 경우**: 단순 CRUD, 기존 패턴을 그대로 반복하는 기능

---

## 파일명 규칙

`tdd-{기능-슬러그}.md` — 예: `tdd-order-payment.md`, `tdd-file-upload.md`

---

## 문서 헤더

```markdown
> 작성일: YYYY-MM-DD
> 상태: Draft | Reviewing | Approved | Superseded
> 대상 모듈: {실제 모듈명}
> 작성자: {이름}
```

---

## 표준 섹션

### 1. 설계 배경 및 목적
- 배경, 설계 목표, 비목표, 기술적 제약

### 2. 현행 시스템 분석
- 관련 도메인 구조, 현재 처리 흐름, 현행 스키마

### 3. 아키텍처 설계
- 계층별 책임 분배 표, 처리 흐름, 설계 대안 분석

### 4. 도메인 모델 설계
- 애그리거트 경계, 도메인 모델 상세, 스키마 변경, 데이터 변환 흐름

### 5. 트랜잭션 설계
- 트랜잭션 경계 표, 정합성 전략, 이벤트 처리

### 6. 예외 및 실패 처리
- 예외 분류 표, 실패 시나리오, 멱등성

### 7. 동시성 및 성능
- 동시성 제어, 성능 고려사항

### 8. 변경 파일 목록
- 경로·모듈·변경유형·설명 표

### 9. 검증 계획
- 시나리오·테스트유형·검증내용·예상결과 표

---

## 7. sample-tdd.md

기존 코드베이스의 대표 기능을 TDD 형식으로 역문서화한다.
가장 복잡하거나 핵심적인 비즈니스 플로우 하나를 선택하여 위의 표준 섹션 구조로 작성한다.

핵심 원칙: **실제로 구현된 코드를 기준으로 설계를 역추론한다.** 없는 설계를 만들어내지 않는다.
